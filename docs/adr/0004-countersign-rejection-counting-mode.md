# ADR-0004：会签驳回采用计数否决模式，条件由业务自定义

## 状态

已接受

## 日期

2026-07-03

## 背景

会签中某个审批人提出驳回意见（即不同意）时，需要明确整个会签的推进规则。该规则直接影响 BPMN 多实例的 `completionCondition` 设计。

## 选项

### 选项 A：一票否决（任一驳回 → 整组退回上一节点）

completionCondition 设置为 `${nrOfCompletedInstances == nrOfInstances}`，会签 API 提供 `rejectCounterSign(taskId, reason)` 方法，调用时直接终止整个多实例并跳回上一节点。

- 适合严格合规审批（如法务审核）
- 过于刚性，非驳回者也被强制退回

### 选项 B：计数否决（驳回按规则汇总，条件由业务自定义）

completionCondition 由业务侧通过表达式自定义，flowable-plus 仅提供：
- 每个会签人员各自的"同意/驳回"作为流程变量记录
- `counterSign(taskId, approved, comment)` 中 `approved` 参数控制投票方向

- 灵活，适配不同审批规则（全票通过/过半数/特定人必须同意等）
- 需要业务侧理解多实例变量模型

### 选项 C：驳回者独立退回（部分人走完、部分人退回）

驳回者单独创建回退路径，同意者继续后续节点。技术上会导致执行路径分裂，状态管理极其复杂。

- 仅限极特殊场景，不通用

## 决策

**选择选项 B（计数否决，条件由业务自定义）**。

### 理由

1. **灵活度最高**：completionCondition 从 `nrOfCompletedInstances == nrOfInstances` 到复杂表达式，全由业务侧控制
2. **最接近中式审批习惯**："全部通过"和"全部驳回"都是明确的，中间态（部分通过部分驳回）需要业务规则界定
3. **选项 A 太刚性**：实际业务中极少要求"一票否决"，可通过 B 的表达式配置实现等效效果
4. **选项 C 过于复杂**：会导致 BPMN 拓扑分裂，不符合会签语义

## 后果

- **正面**：适配各种审批规则，不做预设选择
- **负面**：`completionCondition` 表达式编写有一定学习成本
- **缓解**：flowable-plus 在文档中提供常见规则的表达式模板（全票通过、过半通过、一票否决等）作为参考
